# Mailer Specification — redmine_mail_digest

## 1. Mailer Class

**Class name**: `IssueDigestMailer`  
**File**: `app/mailers/issue_digest_mailer.rb`  
**Inherits from**: `ActionMailer::Base`

**Why not inherit from Redmine's `Mailer`?**
Redmine's `Mailer` class is tightly coupled to its event notification system. Inheriting
from it would pull in notification preferences, tracking, and other behaviors that are
inappropriate for digest emails. `ActionMailer::Base` is the correct base class.

**Configuration**: The mailer uses the same SMTP settings as Redmine (`ActionMailer::Base.smtp_settings`),
which are configured globally in `config/configuration.yml` or via environment variables.

---

## 2. Mailer Action

```ruby
def digest_email(rule, user, issues, grouped_issues)
```

### Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `rule` | `IssueDigestRule` | The digest rule being executed |
| `user` | `User` | The recipient (Redmine User object) |
| `issues` | `ActiveRecord::Relation<Issue>` | The resolved issues for this recipient (already loaded) |
| `grouped_issues` | `Hash` or `Array` | Issues grouped by `rule.group_by`; nil if group_by = none |

### Behavior

1. Build subject line from `rule.email_subject` template or default.
2. Set `to:` to `user.mail`.
3. Set `from:` to Redmine's configured `mail_from` setting.
4. Render HTML part and text part.
5. Return the mail object (caller delivers with `.deliver_now`).

---

## 3. Subject Format

### Default subject template

```
[{project}] {rule_name} — {date}
```

Example:
```
[My Project] Daily open issues — 2026-05-27
```

### Custom subject template

When `rule.email_subject` is set, it is used as a template with token substitution:

| Token | Replaced with |
|-------|--------------|
| `{project}` | `rule.project.name` |
| `{rule_name}` | `rule.name` |
| `{date}` | Current date formatted as `%Y-%m-%d` |
| `{issues_count}` | Number of issues in this email |

**Security**: Template tokens are allowlisted; no dynamic code evaluation. The template is
processed by simple string substitution (`String#gsub`), not `eval` or ERB.

### Subject truncation

If the resulting subject exceeds 255 characters, truncate at 255 with no ellipsis
(email clients handle long subjects; just cap the DB field length).

---

## 4. Sender / From Address

```ruby
from Setting.mail_from
```

Redmine's global "From" address setting is used. This ensures digest emails appear
to come from the same address as other Redmine notifications.

---

## 5. Plain Text Email Format

```
==================================================
{project_name} — {rule_name}
{date_long}
==================================================

{intro_text}
(shown only if rule.email_intro is set)

ISSUES ({issues_count} total)
==================================================

{if grouped}
--- {group_label}: {group_name} ({group_count} issues) ---
{/if}

#{id}  [{tracker}]  {subject}
       Status: {status}  Priority: {priority}  Assignee: {assignee}
       Due: {due_date}  Updated: {updated_on}
       {url}

{repeat for each issue}

==================================================
You received this digest because you are a member of {project_name}.
Manage digest settings: {settings_url}
This digest was generated by Redmine.
==================================================
```

### Issue row fields (plain text)

| Field | Source |
|-------|--------|
| `#id` | `issue.id` |
| `[tracker]` | `issue.tracker.name` |
| Subject | `issue.subject` |
| Status | `issue.status.name` |
| Priority | `issue.priority.name` |
| Assignee | `issue.assigned_to.try(:name)` or "Unassigned" |
| Due date | `issue.due_date` formatted `%Y-%m-%d` or "—" |
| Updated on | `issue.updated_on` formatted `%Y-%m-%d` |
| URL | Full URL to issue: `{base_url}/issues/{id}` |

## 6. HTML Email Format

Follows Redmine's existing HTML email convention. The existing Redmine Roadie Rails
integration handles CSS inlining (converts `<style>` blocks to inline `style=""` attributes).

### HTML structure

```html
<!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8">
  <style>
    /* Table styles matching Redmine's HTML email conventions */
    table { border-collapse: collapse; width: 100%; }
    th { background: #f5f5f5; text-align: left; padding: 6px 8px; }
    td { padding: 6px 8px; border-bottom: 1px solid #e0e0e0; }
    .priority-high { color: #cc0000; }
    .priority-immediate { color: #990000; font-weight: bold; }
    .overdue { color: #cc0000; }
    .badge-status { /* ... */ }
  </style>
</head>
<body>
  <div class="email-header">
    <h2>[Project Name] — Rule Name</h2>
    <p>Date: 2026-05-27</p>
  </div>

  <!-- Optional intro text -->
  <div class="email-intro">
    <p>{intro_text}</p>
  </div>

  <!-- Issue count summary -->
  <p><strong>{issues_count} issues match your digest for {project_name}.</strong></p>

  <!-- Issue table (ungrouped or per group) -->
  <table>
    <thead>
      <tr>
        <th>#</th>
        <th>Tracker</th>
        <th>Subject</th>
        <th>Status</th>
        <th>Priority</th>
        <th>Assignee</th>
        <th>Due date</th>
        <th>Updated</th>
      </tr>
    </thead>
    <tbody>
      <!-- One row per issue -->
      <tr>
        <td><a href="{issue_url}">{issue.id}</a></td>
        <td>{tracker.name}</td>
        <td><a href="{issue_url}">{issue.subject}</a></td>
        <td>{status.name}</td>
        <td class="priority-{priority.css_class}">{priority.name}</td>
        <td>{assigned_to.name}</td>
        <td class="{overdue_class}">{due_date}</td>
        <td>{updated_on}</td>
      </tr>
    </tbody>
  </table>

  <!-- Grouped header (if group_by != none) -->
  <h3>{group_label}: {group_name} ({count} issues)</h3>
  <!-- ... followed by table -->

  <!-- Footer -->
  <div class="email-footer">
    <p>
      You received this digest because you are a member of {project_name}.<br>
      <a href="{settings_url}">Manage digest settings</a><br>
      This digest was generated by Redmine.
    </p>
  </div>
</body>
</html>
```

### Grouped layout

When `rule.group_by != 'none'`:
- Render one `<h3>` header per group followed by a `<table>` for that group.
- Groups are sorted as specified in functional spec section 9.
- Empty groups are omitted.

## 7. Issue Table Columns

| Column | Always shown | Notes |
|--------|-------------|-------|
| `#` (id) | Yes | Linked to issue URL |
| Tracker | Yes | |
| Subject | Yes | Linked to issue URL |
| Status | Yes | |
| Priority | Yes | CSS class for color coding |
| Assignee | Yes | "Unassigned" if nil |
| Due date | Yes | CSS class `overdue` if overdue |
| Updated | Yes | Date only (not time) |

Optional additional columns (not in v1; listed for future reference):
- Estimated hours
- Done ratio
- Category
- Version

## 8. Issue Links

All issue links use the absolute URL: `{Redmine base URL}/issues/{issue.id}`.

**Base URL source**: `Setting.host_name` + `Setting.protocol` (or `Setting.base_url` if set).

Use Redmine's `url_for` with `host:` option, or construct manually:
```ruby
"#{Setting.protocol}://#{Setting.host_name}/issues/#{issue.id}"
```

Avoid using `request` object in the mailer (not available in rake context).

---

## 9. Visibility Filtering

Issue content is already filtered by `Issue.visible(user)` in `IssueResolver`.
The mailer receives only visible issues and renders them all. No additional visibility
check is needed in the mailer itself.

---

## 10. Personalization

The mailer receives only this recipient's issues. There is no generic "all issues" email
that is sent to everyone. Each call to `IssueDigestMailer.digest_email(...)` is for
exactly one `user` and exactly their visible issues.

---

## 11. Handling Empty Digests

- If `issues` is empty and `rule.send_empty == false`: the mailer is **never called**.
  The `DigestSender` skips the recipient (records a `skipped` delivery). No email generated.
- If `issues` is empty and `rule.send_empty == true`: the mailer is called with an empty
  `issues` collection. The email body reads:
  ```
  No issues matched this digest.
  ```
  The issue table is omitted.

---

## 12. Maximum Issues Cap

When `IssueResolver` applies `limit(max_issues_per_email)`, the mailer receives a
pre-limited collection. If the collection was limited:
- A note appears below the issue table:
  ```
  Showing first {max} of {total} total matching issues. 
  View all issues: {project_issues_url}
  ```
- "Total" count is passed explicitly to the mailer as a parameter (the resolver runs a
  separate `.count` before applying the limit, only if needed).

## 16. Multipart Email

The mailer sends a multipart email with both HTML and plain text parts:

```ruby
mail(
  to:      user.mail,
  from:    Setting.mail_from,
  subject: computed_subject
) do |format|
  format.html { render 'digest_email' }
  format.text { render 'digest_email' }
end
```

This matches Redmine's own mailer pattern for HTML+text emails.

---

## 17. Delivery Method

The `DigestSender` calls `.deliver_now` on the mail object:

```ruby
IssueDigestMailer.digest_email(rule, user, issues, grouped_issues).deliver_now
```

`deliver_later` is not used because:
- No background job infrastructure is required.
- The rake task runs synchronously and records results immediately.
- `deliver_now` errors are catchable per-recipient.

If a future version adds background job support, this can be changed to `deliver_later`.
